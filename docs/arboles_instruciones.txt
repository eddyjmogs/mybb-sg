================================================================
 DOJO SHINOBI — Especificación de desarrollo
================================================================
Basado en docs/arboles.txt + aclaraciones del cliente.
Incluye las 4 mejoras de diseño acordadas:
  (1) Estado DERIVADO (no duplicar el árbol por ficha).
  (2) Catálogo de técnicas CACHEADO (datacache de MyBB).
  (3) Una sola función de estado + un solo endpoint de acción.
  (4) Acción ATÓMICA de 1 clic (crédito-primero, si no, cobra Tobis).

----------------------------------------------------------------
1. RESUMEN
----------------------------------------------------------------
El "Dojo Shinobi" es la página donde el personaje GASTA TOBIS para
desarrollar su árbol de técnicas. Acciones:

  CON COSTO (Tobis, salvo que haya crédito gratis):
    A) Desbloquear un árbol (elemental)
    B) Desbloquear una rama
    C) Subir de nivel una rama (aprender 2 mejoras)

  GRATIS (0 Tobis):
    D) Aprender especializaciones (al alcanzar nivel de árbol 3/6/9)
    E) Aprender 1 rama del clan (una sola vez, a elección)

----------------------------------------------------------------
2. PRINCIPIO DE DISEÑO: ESTADO DERIVADO (no duplicar)  [Mejora 1]
----------------------------------------------------------------
El árbol del personaje NO se guarda como un JSON espejo por ficha.
Todo el estado del árbol es una FUNCIÓN PURA de:

    tec_aprendidas (lo que el personaje posee)  ∩  catálogo (lo que existe)

Equivalencias derivadas:
    ¿Tiene el árbol X?      -> posee la base "X101B" en tec_aprendidas
    ¿Rama desbloqueada?     -> posee la BASE de esa rama (según catálogo)
    Nivel de una rama       -> (nº de MEJORAS de esa rama que posee) / 2
    Especializaciones       -> las ESPECIALIDADES de ese árbol que posee

=> Se ELIMINA / IGNORA la columna `mybb_sg_sg_fichas.arboles`.
   Ventaja: una sola fuente de verdad (tec_aprendidas), sin "drift",
   sin doble escritura. Cada acción solo hace:
       INSERT en tec_aprendidas  +  JSON_SET sobre arboles_progreso.

----------------------------------------------------------------
3. DATOS QUE SÍ SE PERSISTEN
----------------------------------------------------------------
- mybb_sg_sg_fichas.tobi ..... moneda que se gasta.
- mybb_sg_sg_fichas.nivel .... nivel del personaje (traba para árboles).
- mybb_sg_sg_fichas.slot_elementales ... INT, default 5. Slots para la ruleta de
  elementos NATURALES (katon/fuuton/suiton/doton/raiton). Cada tirada gasta 1.

- mybb_sg_sg_tec_aprendidas (tid, uid) ... FUENTE DE VERDAD. Cada
  desbloqueo = INSERT aquí.

- mybb_sg_sg_fichas.arboles_progreso (JSON) ... lo ÚNICO no derivable:
      desbloqueo_arboles ....... veces que se PAGÓ por un árbol (escala costo).
      desbloqueo_ramas ......... veces que se PAGÓ por una rama (escala costo).
      desbloqueo_nivel_ramas ... veces que se PAGÓ por subir nivel (escala costo).
      arboles_disponibles ...... (en desuso) los árboles SIEMPRE cuestan Tobis;
                                 se mantiene por compatibilidad. Default 1.
      ramas_disponibles ........ créditos gratis para rama (default 0; staff puede regalar).
      nivel_rama_disponibles ... créditos gratis para subir nivel (default 0; staff puede regalar).
      clan_rama_usada .......... 0/1, si ya se usó la rama de clan gratis.

  (Los créditos permiten "banca": el staff puede regalarlos o el sistema
   los otorga, p. ej. el árbol inicial gratis. Ver sección 7.)

----------------------------------------------------------------
4. CATÁLOGO CACHEADO  [Mejora 2]
----------------------------------------------------------------
El catálogo (qué árboles/ramas/mejoras/especialidades existen) es ESTÁTICO:
solo cambia cuando el staff edita técnicas. NO se consulta por request.

- Se guarda en el datacache de MyBB con clave 'sg_arboles':
      $cache->read('sg_arboles');                  // leer
      $cache->update('sg_arboles', $catalogo);     // reconstruir
- Estructura (lo que ya produce sg_build_arbol por cada árbol):
      catalogo[arbol] = {
        "base": "<tid base del árbol>",
        "rama 1": { "base":"<tid>", "mejoras":[6 tids], "especialidades":[3 tids] },
        "rama 2": { ... },
        "rama 3": { ... }
      }
- Se reconstruye (update) cuando se crea/edita/borra una técnica
  (en crear_tecnicas / modificar_tecnicas), o con un endpoint admin
  (reutilizar generar_arboles.php para poblar la cache).

Resultado: la página corre con ~1 query (tec_aprendidas del uid) + 1 lectura
de cache, sin importar cuántos árboles tenga el personaje.

----------------------------------------------------------------
5. CONCEPTOS DE NIVEL (derivados)
----------------------------------------------------------------
- Una RAMA tiene 6 mejoras = 3 NIVELES (2 mejoras por nivel).
      nivel_de_rama  = (mejoras de esa rama que posee) / 2     -> 0..3
- NIVEL DE ÁRBOL = suma de los niveles de sus 3 ramas          -> 0..9
- Cada rama tiene 3 especializaciones -> 9 por árbol.
- No se guarda ningún "número de nivel": el LARGO del set de mejoras ES el nivel.

----------------------------------------------------------------
6. CÓMO SE DERIVAN LOS "DISPONIBLES"
----------------------------------------------------------------
Partiendo de: owned = set de tids en tec_aprendidas del uid; catalogo (cache).

(6.1) Árboles que posee:
      arbol poseído  <=>  catalogo[arbol].base ∈ owned

(6.2) Árboles elementales adquiribles:
      {katon,suiton,doton,fuuton,raiton,yin,yang}  −  {árboles ya poseídos}

(6.3) Ramas desbloqueables (de árboles poseídos):
      rama desbloqueable  <=>  catalogo[arbol][rama].base ∉ owned

(6.4) Nivel de cada rama desbloqueada:
      nivel = count( catalogo[arbol][rama].mejoras ∩ owned ) / 2

(6.5) Mejoras elegibles para subir nivel (rama desbloqueada):
      libres = catalogo[arbol][rama].mejoras − owned
      elegible para subir nivel  <=>  count(libres) >= 2

(6.6) Especializaciones:
      pool        = ∪ catalogo[arbol][*].especialidades        (9)
      aprendidas  = pool ∩ owned
      cupo        = min( floor(nivel_arbol / 3), 3 )            (0..3)
      elegibles   = pool − aprendidas   (si count(aprendidas) < cupo)

----------------------------------------------------------------
7. MODELO DE ACCIÓN ATÓMICA  [Mejora 4]
----------------------------------------------------------------
Las acciones con costo (A/B/C) son de UN SOLO CLIC y atómicas.

ÁRBOLES (A): SIEMPRE cuestan Tobis (no hay crédito gratis). Ver sección 9.

RAMAS y NIVELES (B/C): regla crédito-primero:
    Si el CRÉDITO correspondiente > 0 -> se consume 1 crédito (GRATIS), crédito -= 1.
    Si el crédito == 0 -> se cobran los Tobis (fórmula) y el CONTADOR += 1.
  (Los créditos de B/C empiezan en 0; el staff puede regalarlos.)

Mapa contador / crédito:
     B) desbloqueo_ramas        <-> ramas_disponibles
     C) desbloqueo_nivel_ramas  <-> nivel_rama_disponibles

----------------------------------------------------------------
8. FÓRMULAS DE COSTO (en Tobis)
----------------------------------------------------------------
  A) Desbloquear ÁRBOL    costo = 15 + (10 × desbloqueo_arboles)      -> 15, 25, 35, 45, 55
  B) Desbloquear RAMA     costo = 10 + ( 5 × desbloqueo_ramas)        -> 10, 15, 20, 25, ...
  C) Subir NIVEL de rama  costo =  5 + ( 2 × desbloqueo_nivel_ramas)  ->  5,  7,  9, 11, ...

----------------------------------------------------------------
9. FEATURES (detalle)
----------------------------------------------------------------

    Los árboles elementales se dividen en DOS grupos:
      NATURALES (katon, fuuton, suiton, doton, raiton) -> solo por RULETA (A2).
      DIRECTOS  (yin, yang)                            -> selección directa (A1).

    Economía COMPARTIDA "desbloquear árbol" (la usan A1 y A2): SIEMPRE cuesta Tobis.
      Requiere nivel + Tobis (fórmula A); luego desbloqueo_arboles += 1:
            desbloqueo_arboles == 0 -> requiere nivel >= 1   (costo 15)  <- el 1º NO es gratis
            desbloqueo_arboles == 1 -> requiere nivel >= 5   (costo 25)
            desbloqueo_arboles == 2 -> requiere nivel >= 10  (costo 35)
            desbloqueo_arboles == 3 -> requiere nivel >= 15  (costo 45)
            desbloqueo_arboles == 4 -> requiere nivel >= 20  (costo 55)

(A1) DESBLOQUEAR UN ÁRBOL DIRECTO (yin / yang)
    Elige yin o yang (si está en catálogo y no lo posees).
    Aplica la economía compartida de arriba.
    Efecto: INSERT de catalogo[arbol].base en tec_aprendidas.

(A2) RULETA ELEMENTAL (naturales, aleatorio + 1 slot)
    Botón "Desbloquea un árbol elemental".
    Requisitos para girar:
      - slot_elementales > 0, Y
      - queda al menos 1 natural bloqueado, Y
      - se cumple la economía compartida (crédito gratis, o nivel + Tobis).
    Al girar:
      - Aplica la economía compartida (crédito o Tobis + desbloqueo_arboles++).
      - Elige AL AZAR uno de los naturales aún bloqueados (el jugador NO escoge).
      - slot_elementales -= 1.
      - Efecto: INSERT de catalogo[elegido].base en tec_aprendidas.
    La UI muestra una ruleta animada que aterriza en el elemento obtenido.

(B) DESBLOQUEAR UNA RAMA
    Elige una rama desbloqueable (6.3) de cualquier árbol poseído.
    - Si ramas_disponibles > 0 -> gratis, ramas_disponibles -= 1.
    - Si no -> cobra fórmula B, desbloqueo_ramas += 1.
    Efecto: INSERT de catalogo[arbol][rama].base en tec_aprendidas.

(C) SUBIR NIVEL DE UNA RAMA (aprender 2 mejoras)
    Elige una rama YA desbloqueada con >= 2 mejoras libres (6.5) y elige 2 de
    esas mejoras libres (de ESA misma rama).
    - Si nivel_rama_disponibles > 0 -> gratis, nivel_rama_disponibles -= 1.
    - Si no -> cobra fórmula C, desbloqueo_nivel_ramas += 1.
    Efecto: INSERT de los 2 tids de mejora en tec_aprendidas.
    (6 mejoras por rama = 3 niveles; si ya tiene las 6, no es elegible.)

(D) APRENDER ESPECIALIZACIONES (gratis)
    Cupo = min(floor(nivel_arbol/3), 3); puedes aprender mientras
    aprendidas < cupo, eligiendo cualquiera del pool de 9 no aprendida (6.6).
    Efecto: INSERT del tid de especialización en tec_aprendidas.
    No toca arboles_progreso (el cupo se deriva del nivel del árbol).

(E) APRENDER 1 RAMA DEL CLAN (gratis, una sola vez)
    El árbol del clan = el que NO es bukijutsu/defensivo/resistencia/taijutsu.
    Solo si clan_rama_usada == 0: elige UNA rama desbloqueable del árbol de clan.
    Efecto: INSERT de la base de esa rama en tec_aprendidas + clan_rama_usada = 1.

----------------------------------------------------------------
10. FUNCIÓN ÚNICA DE ESTADO: sg_dojo_estado()  [Mejora 3]
----------------------------------------------------------------
Una sola función calcula TODO lo derivado; la usan el RENDER y la VALIDACIÓN.

  function sg_dojo_estado($db, $uid):
    1. Lee ficha: tobi, nivel, arboles_progreso (JSON).
    2. Lee owned = tids en tec_aprendidas del uid (1 query).
    3. catalogo = $cache->read('sg_arboles').
    4. Determina árboles poseídos (6.1) y deriva por cada uno:
       ramas (desbloqueada?, nivel, mejoras_libres), nivel_arbol, especializaciones.
    5. Calcula costos actuales (sección 8) y adquiribles (6.2).
    Devuelve algo como:
      [
        'tobi' => ..., 'nivel' => ...,
        'progreso' => [...],
        'costos' => ['arbol'=>15,'rama'=>10,'nivel'=>5],
        'arboles_adquiribles' => ['katon', ...],
        'arboles' => [
           'taijutsu' => [
              'nivel_arbol' => 5,
              'ramas' => [
                 'rama 1' => ['desbloqueada'=>true,'nivel'=>2,'mejoras_libres'=>[...],'puede_subir'=>true],
                 'rama 2' => ['desbloqueada'=>false, ...],
              ],
              'especializaciones' => ['cupo'=>1,'aprendidas'=>0,'elegibles'=>[...]],
           ],
        ],
        'clan' => ['arbol'=>'hoshigaki','rama_gratis_disponible'=> (clan_rama_usada==0)],
      ]

----------------------------------------------------------------
11. ENDPOINT ÚNICO DE ACCIÓN: dojo_accion.php  [Mejora 3]
----------------------------------------------------------------
Un solo archivo maneja todas las acciones con un parámetro `action`:
    action ∈ { arbol, rama, nivel, especializacion, rama_clan }
    + parámetros (arbol, rama, tids elegidos según el caso).

Flujo del endpoint:
    1. GET_LOCK('dojo_<uid>', timeout).
    2. estado = sg_dojo_estado($db, $uid)   // FRESCO dentro del lock.
    3. Verifica que lo pedido está en la lista de ELEGIBLES del estado
       (árbol adquirible / rama desbloqueable / 2 mejoras libres de esa rama /
        especialización dentro de cupo / rama de clan no usada).
    4. Aplica la regla atómica (sección 7): consume crédito o cobra Tobis.
    5. INSERT en tec_aprendidas + JSON_SET sobre arboles_progreso (+tobi).
    6. RELEASE_LOCK. Devuelve estado actualizado (para refrescar la UI).

----------------------------------------------------------------
12. VALIDACIÓN / CONCURRENCIA
----------------------------------------------------------------
- TODA decisión se toma en el servidor con `sg_dojo_estado` recalculado dentro
  del GET_LOCK (nunca confiar en datos del cliente).
- Verifica: Tobis suficientes, traba de nivel (A), crédito>0 vs cobro, que el
  tid/rama elegido esté en la lista de elegibles y no repetido.
- tec_aprendidas usa INSERT IGNORE (uid,tid) para idempotencia.

----------------------------------------------------------------
13. TOPES
----------------------------------------------------------------
- No hay tope artificial: cada cosa solo se puede desbloquear si REALMENTE queda
  algo por desbloquear (árbol no poseído, rama con base no poseída, rama con >=2
  mejoras libres, especialización dentro de cupo).
- La traba de NIVEL del personaje (sección 9.A) limita cuándo se pueden comprar
  más árboles.
